# Data Act Runbook

This runbook translates the EU Data Act into DEALIoT technical controls for connected-product and
related-service data. It is a technical control baseline, not a legal certification or final legal
assessment.

DEALIoT collects data for scientific research. A deployment can still need a Data Act access path
when users are entitled to data generated by connected devices or related services, and when those
users ask for direct access or sharing with a third party such as an application provider or a
research organisation.

## Scope Decision

Before production use, identify each connected product, related service and user role:

- connected telemetry devices, gateways and WildFi ingestion;
- related services that process generated telemetry, media metadata or latest state;
- users such as device owners, field operators, research sponsors or data holders;
- third parties selected by users, including applications and research organisations.

Record this inventory in `dataact.product.catalog` before granting access.

## Architecture Controls

DEALIoT uses dedicated Data Act evidence topics:

| Topic | Evidence |
|---|---|
| `dataact.product.catalog` | connected product, related service, generated data and user roles |
| `dataact.user.access.requests` | user entitlement, scope, time range, purpose and decision |
| `dataact.third_party.sharing` | user-authorized recipient, purpose, scope and transfer status |
| `dataact.user.exports` | export format, delivery channel, object references and completion status |
| `dataact.safeguards` | security, privacy, trade-secret and misuse safeguards |
| `dataact.legal_basis.checks` | GDPR legal-basis check for personal data in Data Act releases |

These topics hold governance metadata only. Raw telemetry, GPS, media or research payloads stay in
the restricted ingestion, processing or object-storage planes.

## Access Model

Default policy:

| Consumer | Default access | Raw access |
|---|---|---|
| Entitled user | direct secure access or minimised export | scoped after entitlement check |
| User-selected application | purpose-bound transfer package | denied unless explicitly approved |
| Scientist or research organisation | DGA-mediated research data product | exceptional and project-bound |

Use the following release order:

1. Register the product and generated data in `dataact.product.catalog`.
2. Capture the user request in `dataact.user.access.requests`.
3. Check entitlement, requested fields, time window and purpose.
4. If personal data is in scope, record the legal-basis check in
   `dataact.legal_basis.checks`.
5. Apply privacy, security, trade-secret and misuse safeguards.
6. Deliver data through secure API, controlled object access or a minimised export.
7. If the user selected a third party, log the transfer in `dataact.third_party.sharing`.
8. Log the export in `dataact.user.exports` and the mediated activity in
   `governance.intermediation.log`.

## Intermediation With Research

When an application or scientist sits between collected data and downstream processing:

- keep Data Act user entitlement evidence separate from DGA research governance evidence;
- combine `dataact.user.access.requests` with `governance.access.requests`;
- require a research project record before scientific access;
- prefer derived, pseudonymised or minimised datasets over raw GPS, raw payloads and media;
- stop transfer or access when the user authorization is withdrawn or expires.

## Operating Rules

- Do not expose unrestricted raw Kafka topics as the default Data Act access path.
- Publish the data categories, format, access method and restrictions for each product.
- Keep terms for third-party sharing transparent and review them for unfair clauses.
- Release data in explicit, portable JSON contracts where possible.
- Record safeguards before refusing or limiting access because of security or trade-secret risk.
- Log denials with a reason, reviewer and remediation path.
- Keep production delivery behind TLS, authentication and least-privilege credentials.
- Ensure recipients cannot expand field scope, time window or purpose without a new request.

## Release Gate

Before enabling Data Act sharing:

1. Review all Data Act products in the management console.
2. Confirm every product has generated data, related services, user roles and format.
3. Test direct user access, denial, export and third-party transfer workflows.
4. Confirm safeguards are documented and proportionate.
5. Confirm research access also passes the DGA research project gate.
6. Export the Data Act evidence topics and attach them to the release record.
