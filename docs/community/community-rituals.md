# Community Rituals

These rituals keep the user community useful, predictable, and maintainable.

## Weekly Triage

Run once per week while the project is actively receiving questions.

Checklist:

- Review new discussions and issues.
- Move unclear issues back to discussion when acceptance criteria are missing.
- Label confirmed bugs, documentation gaps, user feedback, demos, and integrations.
- Identify repeated questions that need documentation.
- Close stale items only with a clear next step or reason.

## Office Hours Agenda

Use this agenda for a public walkthrough, webinar, or private pilot session.

| Minute | Topic |
|---|---|
| 0-5 | Use case and participant goals |
| 5-15 | DEALIoT architecture overview |
| 15-30 | Local smoke test or recorded walkthrough |
| 30-40 | Data contract, governance, and operations discussion |
| 40-50 | Pilot scoring and production gaps |
| 50-60 | Questions, next actions, owners |

## Discussion Moderation Rules

- Keep threads focused on one question or proposal.
- Ask for reproduction details before accepting bug reports.
- Redirect sensitive security or customer context to private channels.
- Convert answers into docs when they are reusable.
- Explain non-goals directly and link to the relevant support boundary.

## Labels To Maintain

Use these labels consistently in issues and pull requests:

- `bug`
- `documentation`
- `enhancement`
- `adoption`
- `demo`
- `user-feedback`
- `integration`
- `good first issue`
- `help wanted`
- `security`

## Escalation

Escalate out of public channels when:

- A vulnerability is involved.
- Production incidents or emergency support are requested.
- Customer data or private endpoints are required to debug.
- Managed hosting, SLOs, or paid support boundaries are being negotiated.
