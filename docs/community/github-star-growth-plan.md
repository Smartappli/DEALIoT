# GitHub Star Growth Plan

This plan turns the adoption and community assets into a repeatable motion for earning legitimate GitHub stars. The objective is not vanity growth. Stars should come from people who understand the architecture, can evaluate it quickly, and may return as users, contributors, pilot teams, or integration partners.

## Positioning

Lead with one clear promise:

```text
DEALIoT is a production-oriented IoT data architecture for moving from MQTT telemetry to governed data products with Kafka, schema governance, stream processing, replay, observability, and compliance evidence.
```

Use this promise consistently in the README, website, launch posts, discussion titles, conference abstracts, and outreach messages.

## Star Conversion Assets

| Asset | Purpose | Owner action |
|---|---|---|
| README opening | Explain the problem and fit in under 30 seconds | Keep the pitch concrete and avoid broad platform claims |
| Website landing page | Give non-GitHub visitors a fast overview | Link directly to repository, quick evaluation, use cases, and discussions |
| Quick evaluation path | Convert curiosity into a technical trial | Keep the first command path short and reproducible |
| Use case catalog | Help visitors self-identify | Add examples from livestock, agriculture, industrial IoT, research, governance, and integrations |
| First GitHub Discussion | Show that the project is alive and wants feedback | Ask one concrete question and point to the quick evaluation path |
| Good first issues | Convert stars into contributors | Keep issues small, labeled, mentored, and testable |
| Public launch kit | Make sharing easy | Reuse the same pitch across LinkedIn, GitHub, meetups, partner emails, and newsletters |
| Adopter story template | Turn real outcomes into proof | Publish only approved and sanitized stories |

## 7-Day Launch Motion

### Day 0: Prepare The Repository

- Confirm README links to website, quick evaluation, use cases, discussions, contribution path, and launch kit.
- Confirm the website has visible links to the repository and community entry points.
- Open at least three `good first issue` tasks with exact scope and validation steps.
- Enable GitHub Discussions if it is not already enabled.
- Prepare the first launch discussion from `docs/community/first-github-discussion.md`.

### Day 1: Publish The First Discussion

- Create the first GitHub Discussion using the prepared title and body.
- Pin or feature it if repository settings allow.
- Ask visitors to reply with one use case, one evaluation blocker, or one integration proposal.
- Ask for a star only after explaining the repository value and evaluation path.

### Day 2: Publish One External Post

- Use the launch post from `docs/community/public-launch-kit.md`.
- Link to the repository, first discussion, quick evaluation path, and use case catalog.
- Prefer a technical audience: IoT engineers, data platform teams, research infrastructure teams, agriculture technology builders, and open-source infrastructure communities.

### Day 3: Share In Targeted Communities

- Share one useful technical angle per community.
- Do not post generic promotion.
- Ask for feedback on a concrete topic: MQTT ingestion, Kafka schemas, Flink processing, replay/backfill, observability, governance evidence, or deployment hardening.
- Record every repeated question as a documentation or test candidate.

### Day 4: Convert Interest Into Work

- Turn repeated questions into issues or documentation patches.
- Label small tasks with `good first issue`, `help wanted`, `mentored`, and an area label.
- Reply to new discussions with links to exact docs or issues.
- Add missing acceptance criteria before inviting contributions.

### Day 5: Publish A Proof Asset

- Share one public-safe demo note, screenshot, smoke-test result, pilot lesson, or architecture walkthrough.
- Do not include secrets, customer data, personal data, or private endpoint values.
- Link back to the repository and quick evaluation path.

### Day 7: Publish A Recap

- Summarize what was learned, what changed in docs or tests, and what help is needed next.
- Thank contributors by GitHub handle only when their contribution is public.
- Link to open mentored issues and the next discussion topic.

## Distribution Channels

Prioritize channels where the audience can evaluate technical infrastructure:

- GitHub Discussions and repository README.
- LinkedIn posts from maintainers and project accounts.
- Dev.to or Hashnode technical walkthrough.
- Hacker News `Show HN` only after the quick evaluation path is reliable.
- Reddit communities only when rules permit project sharing and the post contains technical detail.
- IoT, Kafka, Flink, data engineering, agritech, and research infrastructure Slack or Discord communities.
- University, public-sector, and research infrastructure mailing lists with a concrete evaluation ask.
- Partner emails to device, hosting, data governance, and compliance stakeholders.

## Weekly Operating Loop

Run this loop every week for the first two months:

- Review repository traffic, stars, forks, discussions, issues, and failed evaluation reports.
- Identify the page or command where evaluators stop.
- Ship one improvement to README, quick evaluation, website, tests, examples, or discussion answers.
- Promote one concrete proof asset, not the same generic announcement.
- Add or refresh two mentored good-first issues.
- Close the loop publicly when a user question becomes documentation or a test.

## Metrics

Track quality before volume:

- Stars per launch post.
- Clicks from website to repository.
- Discussions opened by external users.
- Clone-to-smoke-test success reports.
- Good-first issues opened, assigned, and merged.
- Use cases proposed.
- Pilot requests with one concrete data source.
- Public or private adopter stories approved for reference.

## Do Not Do

- Do not buy stars, run star-for-star exchanges, or ask unrelated communities for empty support.
- Do not spam communities with the same message.
- Do not claim adoption without approved adopter evidence.
- Do not overstate compliance status from templates alone.
- Do not invite production users before security, deployment, and support boundaries are clear.
- Do not push users into private channels when the answer belongs in public documentation.
